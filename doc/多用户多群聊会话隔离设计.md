# 多用户 & 多群聊会话隔离设计方案

**版本**: v1.0
**日期**: 2026-04-03
**状态**: 设计评审

---

## 1. 问题背景

### 1.1 当前架构问题

当前设计中，`sessionId` 仅基于 `CLAUDE_PROCESS_NAME` 生成，导致所有钉钉会话（单聊 + 群聊）共享同一个 Claude CLI 进程和会话上下文：

```typescript
// 当前实现 (claude/client.ts:96-99)
constructor(processName: string = 'default') {
  this.processName = processName;
  this.sessionId = generateUUIDFromString(processName);  // ← 问题：全局唯一
}
```

**影响范围**：

| 场景 | 问题描述 | 严重程度 |
|------|----------|----------|
| 单人单聊 | ✅ 无影响，上下文连续 | - |
| 多人单聊 | ⚠️ 用户 A 和用户 B 的对话在 Claude 侧混合 | 高 |
| 多群聊 | ⚠️ 群聊 X 和群聊 Y 的上下文混合 | 高 |
| 隐私敏感场景 | ❌ 不同用户的上下文互相可见 | 严重 |

### 1.2 问题示例

```
时间线：
T1: 用户 A (单聊)  → "帮我分析这个 bug"
T2: 用户 B (单聊)  → "明天天气怎么样"
T3: 群聊 X (群聊)  → "项目进度如何"
T4: 用户 A (单聊)  → "刚才说的那个..."  ← Claude 可能引用用户 B 或群聊 X 的内容！
```

## 2. 设计目标

### 2.1 功能性目标

| 目标 | 描述 | 优先级 |
|------|------|--------|
| 会话隔离 | 每个钉钉 `conversationId` 对应独立的 Claude 会话 | P0 |
| 上下文持久化 | 同一会话内多轮对话上下文连续 | P0 |
| 跨重启保持 | Bot/Proxy 重启后，会话 ID 不变，上下文可恢复 | P0 |
| 资源可控 | 限制并发 Claude 进程数量，避免资源耗尽 | P1 |

### 2.2 非功能性目标

| 目标 | 描述 |
|------|------|
| 向后兼容 | 现有单用户场景无需修改配置 |
| 性能影响小 | 会话切换延迟 < 100ms |
| 可观测性 | 每个会话可追踪、可监控 |

## 3. 设计方案

### 3.1 核心思路

**将会话标识从"全局唯一"改为"按 conversationId 隔离"**：

```
┌─────────────────────────────────────────────────────────────┐
│                      修改前                                  │
├─────────────────────────────────────────────────────────────┤
│  CLAUDE_PROCESS_NAME = "dingtalk-bot"                       │
│          ↓                                                  │
│  sessionId = SHA256("dingtalk-bot") = 固定值                 │
│          ↓                                                  │
│  所有会话共享 → Claude CLI (sessionId: abc123...)           │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                      修改后                                  │
├─────────────────────────────────────────────────────────────┤
│  CLAUDE_PROCESS_NAME = "dingtalk-bot"                       │
│          ↓                                                  │
│  sessionId(conversationId) = SHA256(processName + ":" + conversationId) │
│          ↓                                                  │
│  用户 A (conv: "user123") → Claude CLI (sessionId: xxx...)  │
│  用户 B (conv: "user456") → Claude CLI (sessionId: yyy...)  │
│  群聊 X (conv: "group789") → Claude CLI (sessionId: zzz...) │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 方案选型

#### 方案一：动态 Session-ID（推荐）

**核心思想**：在 `ClaudeClient` 中按 `conversationId` 动态计算 `sessionId`，Proxy 按 `sessionId` 区分不同会话。

```
│  - instances: Map<string, ClaudeInstance>                            │
└──────────────────────────────────────────────────────────────────────┘
                                   ↓

│                           ClaudeInstance                             │
├──────────────────────────────────────────────────────────────────────┤
│  - sessionId: string                                                 │
│  - socket: net.Socket | null                                         │

│  + connect(): Promise<boolean>                                       │
```

**2. 修改 `ClaudeClient` 为实例管理器** (`src/claude/client.ts`)：

```typescript
import { ClaudeInstance } from './instance.js';

export class ClaudeClient {
  private processName: string;
  private instances: Map<string, ClaudeInstance> = new Map();
  private maxInstances: number = 50;  // 可配置

  constructor(processName: string = 'default') {
    this.processName = processName;
  }

  /**
   * 获取或创建指定 conversationId 的 Claude 实例
   */
  getInstance(conversationId: string): ClaudeInstance {
    if (!this.instances.has(conversationId)) {
      // 检查是否达到上限
      if (this.instances.size >= this.maxInstances) {
        // 简单的 LRU 策略：删除最旧的实例
        const oldestKey = this.instances.keys().next().value;
        if (oldestKey) {
          this.instances.get(oldestKey)?.disconnect();
          this.instances.delete(oldestKey);
          logger.info('Claude-Code', 'Evicted oldest instance', { conversationId: oldestKey });
        }
      }

      const instance = new ClaudeInstance(conversationId, this.processName);
      this.instances.set(conversationId, instance);
    }
    return this.instances.get(conversationId)!;
  }

  /**
   * 断开指定会话
   */
  disconnect(conversationId?: string) {
    if (conversationId) {
      this.instances.get(conversationId)?.disconnect();
      this.instances.delete(conversationId);
    } else {
      // 断开所有
      for (const instance of this.instances.values()) {
        instance.disconnect();
      }
      this.instances.clear();
    }
  }

  /**
   * 获取实例统计信息
   */
  getStats() {
    return {
      processName: this.processName,
      activeInstances: this.instances.size,
      maxInstances: this.maxInstances,
      conversationIds: Array.from(this.instances.keys()),
    };
  }
}
```

#### 方案二：集中式会话管理器（备选）

**核心思想**：由 Proxy 统一管理多个 Claude CLI 进程，Bot 只需传递 `conversationId`。

```
┌─────────────────────────────────────────────────────────────┐
│                        Bot                                  │
│  发送：{ type: "user", conversationId: "xxx", content: "..." } │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│                        Proxy                                │
├─────────────────────────────────────────────────────────────┤
│  SessionManager:                                            │
│  - instances: Map<sessionId, ClaudeProcess>                │
│                                                             │
│  onMessage(data):                                           │
│    sessionId = hash(data.conversationId)                   │
│    instance = getOrCreate(sessionId)                        │
│    instance.send(data.content)                              │
└─────────────────────────────────────────────────────────────┘
```

**优缺点对比**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 方案一（推荐） | 改动小，逻辑清晰，易于调试 | Bot 需要管理多个实例 |
| 方案二（备选） | Bot 逻辑简单，Proxy 集中管理 | Proxy 复杂度增加，调试困难 |

### 3.3 会话生命周期管理

#### 会话创建

```
用户消息到达
    ↓
Bot.getInstance(conversationId)
    ↓
检查 instances Map
    ↓
不存在 → 创建新 ClaudeInstance
存在 → 复用已有实例
    ↓
instance.connect() → 连接对应 Proxy
```

#### 会话复用

```
同一 conversationId 的后续消息
    ↓
getInstance(conversationId) 返回已有实例
    ↓
复用同一 Claude CLI 进程
    ↓
上下文连续 ✅
```

#### 会话清理

**策略一：LRU 淘汰**
```typescript
if (this.instances.size >= this.maxInstances) {
  const oldestKey = this.instances.keys().next().value;
  this.instances.get(oldestKey)?.disconnect();
  this.instances.delete(oldestKey);
}
```

**策略二：空闲超时**
```typescript
// 每个实例记录最后活跃时间
private lastActiveAt: number = Date.now();

// 定期清理超过 30 分钟未活跃的实例
setInterval(() => {
  const now = Date.now();
  for (const [key, instance] of this.instances) {
    if (now - instance.lastActiveAt > 30 * 60 * 1000) {
      instance.disconnect();
      this.instances.delete(key);
    }
  }
}, 5 * 60 * 1000);  // 每 5 分钟检查一次
```

## 6. 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 资源消耗增加 | 每个会话一个 Claude 进程，内存/CPU 增加 | 设置 `CLAUDE_MAX_INSTANCES` 限制 |
| Proxy 连接数增加 | Named Pipe 并发连接数上限 | 监控连接数，设置合理阈值 |
| 会话状态不一致 | Bot 内存中的实例状态与 Proxy 不一致 | 实现自动重连和状态同步 |
| 向后兼容性 | 现有配置可能不兼容 | 提供默认值，平滑迁移 |

```typescript
function generateSessionId(processName: string, conversationId: string): string {
  const combined = `${processName}:${conversationId}`;
  const hash = createHash('sha256').update(combined).digest('hex');
  return `${hash.substring(0, 8)}-${hash.substring(8, 12)}-4${hash.substring(13, 16)}-${hash.substring(16, 20)}-${hash.substring(20, 32)}`;
}
```
