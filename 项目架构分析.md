# 项目架构分析（3W2H）

## 1. What（是什么）

一个中间件服务，将 **钉钉消息** 转发给本地运行的 **Claude Code CLI**，并将 Claude 的执行过程（工具调用、结果、回复）以 **交互式卡片** 形式实时回传到钉钉。

**核心能力：**
- 接收钉钉文本消息 → 调用 Claude Code
- 流式输出 Claude 的所有事件（Read/Edit/Bash 等工具调用 + 结果 + 回复）
- 支持图片发送（通过 MCP 工具 `dingtalk_send_image`）
- 支持群聊/单聊
- 支持长内容自动分卡

---

## 2. Why（为什么这样设计）

**解决的问题：**

| 问题 | 解决方案 |
|------|----------|
| Claude CLI 是本地交互工具，无法远程调用 | 通过 **Proxy 代理架构** 暴露 IPC 接口 |
| Bot 重启会丢失 Claude 会话上下文 | Proxy 独立运行，Bot 重启不影响 Claude CLI 进程 |
| Claude CLI 崩溃会导致服务中断 | Proxy 自动重启 Claude CLI（指数退避） |
| 长回复超出钉钉卡片限制 | 自动检测长度并分割为多张卡片 |
| 需要实时看到 AI 思考过程 | 流式卡片更新（500ms 防抖） |
| 图片无法直接发送 | MCP 工具 + 钉钉 media upload API |

---

## 3. Who（参与者/组件）

```
┌──────────────┐     ┌───────────────┐     ┌─────────────┐     ┌──────────────┐
│  DingTalk    │────▶│ DingTalkClient │────▶│ DingTalkBot │────▶│ ClaudeClient │
│  WebSocket   │     │ (SDK + API)    │     │ (会话管理)   │     │ (事件解析)   │
└──────────────┘     └───────────────┘     └─────────────┘     └──────┬───────┘
                                                                      │
                        ┌──────────────┐                              │ Named Pipe
                        │ Claude Proxy │◀─────────────────────────────┘
                        │ (独立进程)    │
                        └──────┬───────┘
                               │ stdio
                        ┌──────▼───────┐
                        │  Claude CLI  │
                        │  (用户进程)   │
                        └──────────────┘
```

**组件职责：**

| 组件 | 职责 |
|------|------|
| `src/index.ts` | 入口，组装各组件，优雅停机 |
| `src/dingtalk/client.ts` | 钉钉 WebSocket 连接、卡片 API、Token 缓存 |
| `src/dingtalk/bot.ts` | 消息路由、会话管理、去重、卡片更新逻辑 |
| `src/claude/client.ts` | 连接 Proxy、解析 stream-json 事件、格式化为 Markdown |
| `src/claude/proxy.ts` | 独立进程，管理 Claude CLI 生命周期，暴露 Named Pipe |
| `src/mcp/dingtalk-image-server.ts` | MCP 工具服务，支持发送图片到钉钉 |

## 4. How（如何工作）

**消息处理流程：**

```
1. 钉钉消息 → WebSocket → DWClient
2. DingTalkClient.handleCallback() → 提取 conversationId/text/sender
3. DingTalkBot.handleMessage()
   - 创建/获取会话（50 条历史）
   - 创建流式卡片（"正在思考..."）
   - 写入 .dingtalk-context.json（MCP 工具用）
   - 调用 ClaudeClient.streamMessage()
4. ClaudeClient 通过 Named Pipe 发送消息给 Proxy
5. Proxy 转发给 Claude CLI（stdin）
6. Claude CLI 输出 stream-json 事件（stdout）
7. Proxy 通过 Named Pipe 回传事件
8. ClaudeClient 解析事件：
   - assistant(tool_use) → 格式化为 "🔧 Read `file.txt`"
   - user(tool_result) → 截断展示结果
   - assistant(text) → 透传文本
   - result → 统计信息
9. 累积响应 → 防抖更新卡片（500ms）
10. 流式结束 → Finalize 卡片
```

**Proxy 架构优势：**
```
Bot (可重启) ←Named Pipe→ Proxy (长驻) ←stdio→ Claude CLI (长驻)
```
- Bot 重启只需重连 Pipe，不影响 Claude 会话
- Proxy 自动重启崩溃的 Claude CLI（3s→6s→12s 指数退避）
- 会话 ID 由 `CLAUDE_PROCESS_NAME` 确定性生成，上下文持久化

## 6. 为什么 Bot 和 Proxy 要分离？

### 6.1 核心问题：如果不分离会怎样？

```
【单体架构】
┌──────────────────────────────┐
│         Bot + Claude         │
│  ←WebSocket→  │  ←stdio→     │
│  钉钉         │   Claude CLI │
└──────────────────────────────┘
         ↓ 重启
   Claude CLI 也被杀死 ❌
   正在进行的任务中断 ❌
   会话上下文丢失 ❌
```

### 6.2 分离架构的五大价值

#### 价值一：生命周期解耦

| 场景 | 单体架构 | 分离架构 |
|------|----------|----------|
| Bot 代码更新重启 | Claude CLI 被杀死，任务中断 | Claude CLI 继续运行，任务不受影响 |
| Bot 崩溃 | Claude CLI 也被带走 | Proxy 自动重启 Claude 或等待 Bot 重连 |
| 调试 Bot 代码 | 需要频繁启停，Claude 无法保持会话 | Claude 会话持久化，调试效率大幅提升 |

**通俗理解**：
> Bot 是"前台接待员"，Proxy 是"管家"，Claude CLI 是"专家"。
> 接待员换班（Bot 重启），不应该把专家和管家都赶走。

#### 价值二：会话上下文持久化

```
【分离架构效果】
用户：帮我分析这个项目
Bot: 正在读取文件...
     [Bot 崩溃重启]
用户：继续
Bot: ✅ 继续分析，因为 Claude CLI 一直运行，上下文完整
```

#### 价值三：故障隔离

```
【故障传播链】

单体架构：
钉钉 API 异常 → Bot 崩溃 → Claude CLI 被杀 → 用户任务丢失 ❌

分离架构：
钉钉 API 异常 → Bot 崩溃 → Proxy + Claude CLI 继续运行 → Bot 重启后重连 ✅
```

#### 价值四：资源利用优化

| 维度 | 单体 | 分离 |
|------|------|------|
| Claude CLI 启动开销 | 每次 Bot 启动都要重新初始化 | 只需启动一次，后续 Bot 直接连接 |
| Token 预热 | 每次重新 warming up | 保持热状态，响应更快 |
| 文件句柄/缓存 | 随 Bot 重启丢失 | 持续积累，效率更高 |

#### 价值五：运维灵活性

```bash
# 场景 1：Bot 代码更新，无缝重启
npm run build && systemctl restart dingtalk-bot
# → Proxy 和 Claude CLI 不受影响

# 场景 2：查看 Proxy 日志，诊断问题
tail -f /tmp/claude-proxy-default.log

# 场景 3：手动停止 Claude（需要时）
kill $(cat /tmp/claude-proxy-default.pid)

# 场景 4：多个 Bot 实例共享一个 Claude Proxy
# （通过不同 processName 区分）
```

### 6.3 架构对比

```
┌─────────────────────────────────────────────────────────────────┐
│                      单体架构 vs 分离架构                        │
├─────────────────────────────────────────────────────────────────┤
│  单体架构：                                                      │
│  ┌─────────────────┐                                           │
│  │ Bot + Claude    │  ← 一荣俱荣，一损俱损                       │
│  └─────────────────┘                                           │
│                                                                 │
│  分离架构：                                                      │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐                 │
│  │   Bot    │ ←→ │  Proxy   │ ←→ │  Claude  │                 │
│  │ (可重启) │    │  (长驻)  │    │  (长驻)  │                 │
│  └──────────┘    └──────────┘    └──────────┘                 │
│       ↓                ↓               ↓                        │
│   无状态服务       进程管理         核心计算                      │
│   随便重启         保活 + 重连        上下文持久                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.4 一句话总结

> 分离架构让 Bot 成为"无状态的接线员"，Proxy+Claude 成为"有状态的核心服务"，各司其职，互不拖累。

## 5. How Much（成本/效果）

| 维度 | 设计 |
|------|------|
| **性能** | Token 缓存 2 小时、防抖 500ms、结果截断（25 行/1500 字） |
| **可靠性** | 断线重连、指数退避重启、PID 保活检测 |
| **扩展性** | MCP 工具可插拔、多会话并发、长内容自动分卡 |
| **可维护性** | 结构化日志、分层清晰、配置集中（.env） |
